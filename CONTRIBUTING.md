# Contributing

Thanks for your interest in improving `@tegroton/tegro-finance`. The SDK is intentionally small — a typed read/quote/build client, amount helpers, and a TON Connect adapter, plus types and tests. The contribution bar is therefore higher than for a typical SDK: the goal is to **stay minimal and obviously correct**, not to grow surface area.

## TL;DR

```bash
git clone https://github.com/TegroTON/tegro-finance-sdk.git
cd tegro-finance-sdk
npm ci
npm test
npx tsc --noEmit
```

If those four commands pass on a fresh clone, you have a working dev environment.

## Prerequisites

- **Node.js ≥ 20.** The CI matrix runs on 20, 22, 24. Node 18 is the published floor but is not tested.
- **npm ≥ 10.** Required for `npm ci` against the committed lockfile.
- No global tooling. Everything lives in `devDependencies`. The package has **zero runtime dependencies** — keep it that way.

## Project layout

```
src/
  client.ts      HTTP client: read / quote / build. The only module that does I/O.
  amounts.ts     BigInt-exact units + slippage. No floating-point math on money.
  tonconnect.ts  Maps a built transaction to the wallet's sendTransaction shape.
  types.ts       Request/response types, mirrored from the backend's models.
  index.ts       Public surface.
tests/           Vitest specs. Network is mocked.
examples/        Runnable examples (read, quote+build, TON Connect React).
dist/            Generated by `npm run build` — never committed.
```

## What we accept

- **Bug fixes** with a failing test that demonstrates the bug.
- **New endpoint coverage** when the upstream API exposes a stable endpoint we don't wrap yet (staking, IDO, …). Open an issue first to confirm scope.
- **Documentation** that matches the live HTTP API. Anything that contradicts it gets rejected.
- **Test additions** that close uncovered branches.
- **CI / tooling** improvements that don't add complexity.

## What we don't accept

- **New runtime dependencies** unless absolutely unavoidable. TON Connect stays an *optional peer* dependency, not a hard one.
- **Custom abstractions** on top of the API (retries, queues, "smart" caching, route optimizers beyond what the backend returns). Consumers can wrap the client themselves.
- **Key handling or signing.** This SDK is non-custodial by design — the wallet signs. PRs that introduce private-key handling will be closed.
- **Breaking changes** unless tied to a `0.x → 0.y` bump (pre-1.0) or a major release.

## Development workflow

1. **Open an issue** describing the change. Skip only for trivial typos.
2. **Branch from `main`.** Name it descriptively: `feat/staking-endpoints`, `fix/slippage-rounding`.
3. **Write the test first** for bug fixes — the diff should show it failing on `main`, passing after.
4. **Run the full check locally:**

   ```bash
   npm ci
   npx tsc --noEmit
   npm run build
   npm test
   ```

5. **Update the changelog** under an `## Unreleased` heading at the top of `CHANGELOG.md` (or note in the PR that no entry is needed).
6. **Open the PR.** Fill in the template. Link the issue.

## Commit message style

We use [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/): `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`, `ci:`, `deps:`. Scope is optional: `feat(client): add staking endpoints`.

## Coding conventions

- **TypeScript strict mode** is enforced. Don't relax it.
- **No `any`** in exported types. Prefer `unknown` and narrow.
- **No floating-point arithmetic on token amounts.** Use BigInt via the helpers in `amounts.ts`.
- **No `console.log`** in committed library code (`src/`). Examples may log.
- **Match the surrounding style** — the repo doesn't enforce a formatter.

## Tests

- We use [Vitest](https://vitest.dev/). One spec file per module.
- Network is mocked by injecting `fetch` via the client options. **Don't introduce real HTTP in unit tests.**
- New public methods need at least: happy path, request-shape assertion, and an error path where applicable.
- `npm test` must stay green on Node 20, 22, 24.

## Reporting security issues

**Don't open public issues for security problems.** See [SECURITY.md](SECURITY.md) for the private vulnerability reporting flow.

## Code of conduct

Be patient, be specific, and assume good faith. PRs to a money-adjacent library get extra scrutiny — that's diligence, not unfriendliness.
